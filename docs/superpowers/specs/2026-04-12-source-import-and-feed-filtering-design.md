# 信息源导入与列表筛选增强设计

## 背景

当前系统已经支持后台维护分组与信息源、抓取 RSS 内容、将多条相关内容聚合为事件流，并在前台列表页按时间范围和排序方式浏览内容。但仍存在三个明显缺口：

1. 新增信息源需要手工逐条录入，无法导入常见的 OPML 订阅备份。
2. 管理员输入 RSS 链接时，系统不会自动解析订阅标题与站点地址，需要手工补全。
3. 前台列表页缺少基于分组和信息源的筛选能力，聚合内容也无法只展示命中的相关条目。

本次增强的目标是在不改变现有抓取和聚合主流程的前提下，为后台补齐批量导入与自动补全能力，为前台补齐分组/信息源筛选能力，并让聚合结果在筛选场景下遵循稳定一致的展示语义。

## 目标

- 支持导入 OPML 文件或文本内容。
- 按 OPML 中的分类层级自动创建分组，并把订阅源归属到对应分组。
- 基于 RSS 链接自动解析信息源基础信息，至少补齐名称和站点地址。
- 列表页支持按分组、信息源筛选内容。
- 聚合内容在筛选命中时只展示命中的相关条目。
- 聚合内容在筛选后若仅剩一条命中条目，退化为单条内容样式展示。

## 非目标

- 不支持多级分组在数据库中保留树形结构。本次只取最接近订阅源的父级分类作为分组名。
- 不为 OPML 导入新增复杂的冲突解决 UI。本次采用服务端归一化和幂等 upsert 策略。
- 不调整现有聚类算法、AI 分析提示词或抓取调度机制。
- 不引入额外的第三方数据库或后台任务系统。

## 用户场景

### 场景 1：管理员导入 OPML

管理员在后台上传 OPML 文件后，系统解析其中的订阅节点。若某订阅位于分类节点下，则使用该分类名称创建或复用分组，并把该订阅源挂到对应分组。若该 RSS URL 已存在，则更新已有记录而不是重复创建。

### 场景 2：管理员手动新增单个信息源

管理员输入 RSS URL 后，可触发“根据 RSS 自动填充”。系统读取订阅元信息并回填推荐名称、站点地址。管理员仍可手工修改后再保存。

### 场景 3：用户在前台按分组或信息源筛选

用户选择某个分组或信息源后，列表只展示与筛选条件匹配的内容。若某个聚合组中只有部分条目命中，则聚合卡片只展示这些命中条目；若过滤后只剩一条条目，则直接按单条内容卡片展示。

## 设计方案

### 方案对比

#### 方案 A：查询期按 item 过滤，映射期重建聚合展示

先用 item 级条件筛出命中的内容，再根据所属 cluster 重建前台列表项。单条内容直接映射，聚合内容根据命中的条目数量决定保持聚合还是退化为单条。

优点：

- 语义与用户筛选预期完全一致。
- API 返回的数据就是最终展示态，前后端职责清晰。
- 聚合退化规则、计数、展开内容都能保持一致。

缺点：

- 需要调整 feed repository 的查询与映射逻辑。

#### 方案 B：先查出完整聚合，再在前端二次裁切

优点：

- 后端改动较少。

缺点：

- 分页、排序、计数和展开态都会与真实筛选结果不一致。
- 前端需要承担过多业务逻辑，后续维护风险高。

#### 方案 C：为筛选场景新增独立视图模型

优点：

- 可以和默认列表彻底隔离。

缺点：

- 逻辑重复，当前项目规模下维护成本过高。

最终采用方案 A。

## 架构与模块边界

### 后台设置服务

在 `src/lib/settings/service.ts` 中新增信息源导入与 RSS 解析能力，作为后台页面和 API route 的唯一业务入口。该层负责：

- 解析 OPML 内容。
- 规范化 OPML 节点为分组和信息源输入。
- 调用 RSS 解析工具补齐信息源元数据。
- 执行分组查找/创建、信息源 upsert 与批量导入结果统计。

### RSS 解析工具

在 `src/lib/ingestion/parser.ts` 附近新增或扩展纯函数工具，复用现有 `rss-parser` 能力读取 feed 元信息。该工具只负责从 RSS URL 解析：

- feed 标题
- feed 链接
- 可选 feedUrl

并统一做回退逻辑，不直接操作数据库。

### 前台 feed 查询层

在 `src/lib/feed/types.ts`、`src/lib/feed/range.ts`、`src/lib/feed/repository.ts` 中扩展筛选参数与展示映射。repository 负责：

- 根据时间范围、排序、分组、信息源过滤 item。
- 将命中 item 按 cluster 组织。
- 对 cluster 做“保留聚合”或“退化为单条”的最终判定。

### 页面与 API 层

- `src/app/api/admin/settings/sources/route.ts` 与 `src/app/api/admin/settings/sources/[id]/route.ts` 继续负责单源创建与更新。
- 新增后台 API route 处理：
  - RSS 元信息解析
  - OPML 导入
- `src/app/api/feed/route.ts` 与 `src/app/page.tsx` 扩展查询参数解析。
- `src/components/admin/admin-settings-panel.tsx` 与 `src/components/feed/feed-panel.tsx` 承担新增交互。

## 数据模型设计

### 现有模型复用

当前 Prisma 中已有：

- `SourceGroup`
- `Source`
- `Item`
- `ContentCluster`

这足以承载本次需求，因此不新增表结构。

### 分组映射规则

- OPML 中带 `xmlUrl` 的 outline 节点视为订阅源。
- 若该节点存在父分类，则取离它最近的父分类名称作为 `SourceGroup.name`。
- 若订阅源位于顶层且无分类，则不自动创建分组，`groupId` 置空。
- 若 OPML 出现多个同名分类，本次统一归并到同一个 `SourceGroup`，以名称唯一约束为准。

### 信息源去重规则

- 以 `rssUrl` 作为唯一键。
- 导入时若 `rssUrl` 已存在，则更新以下字段：
  - `name`
  - `siteUrl`
  - `groupId`
  - `enabled`
  - `fetchFullTextWhenMissing`
- 默认不在导入时删除未出现在 OPML 中的旧信息源。

## RSS 元信息解析设计

### 输入

- `rssUrl`

### 输出

- `name`
- `siteUrl`
- `rssUrl`

### 解析顺序

1. 使用 `rss-parser` 读取 feed。
2. 优先采用 feed 自带元信息：
   - `title` 作为名称
   - `link` 作为站点地址
3. 若 `link` 缺失，则尝试从 RSS URL 推断站点根地址。
4. 若 `title` 缺失，则以 RSS URL 的 hostname 作为回退名称。
5. 若 RSS 无法读取或格式异常，则返回错误，由 API route 以 4xx/5xx 反馈前端。

### 归一化规则

- 去除名称与 URL 首尾空白。
- `siteUrl` 必须是合法 URL；若 feed 给出的 `link` 不合法，则回退到 RSS URL 推断值。
- 若 RSS URL 本身非法，则直接拒绝请求。

## OPML 导入设计

### 支持的输入形式

- 后台页面上传 `.opml` 文件。
- API 内部统一转为文本字符串处理。

### 解析规则

- 解析 XML。
- 遍历所有 `outline` 节点。
- 只把带 `xmlUrl` 的节点认定为订阅节点。
- 订阅节点字段来源优先级：
  - `name`: `title` -> `text` -> RSS 自动解析结果 -> hostname
  - `rssUrl`: `xmlUrl`
  - `siteUrl`: `htmlUrl` -> RSS 自动解析结果 -> 由 RSS URL 推断
- 分类名取最近父 outline 的 `title` 或 `text`。

### 写入策略

- 导入前先缓存已有分组和信息源，减少重复查询。
- 对每个订阅源执行：
  - 解析/回填元信息
  - 创建或复用分组
  - 按 `rssUrl` upsert 信息源
- 返回导入摘要：
  - 总订阅数
  - 新建数
  - 更新数
  - 失败数
  - 失败详情列表

### 默认字段

- `enabled = true`
- `fetchFullTextWhenMissing = true`

## 前台筛选设计

### 查询参数

首页和 `/api/feed` 同时支持：

- `range`
- `sort`
- `start`
- `end`
- `groupId`
- `sourceId`

### 筛选优先级

- 时间范围、自定义日期范围、分组、信息源共同生效。
- 若同时传入 `groupId` 和 `sourceId`，则视为交集过滤。
- 若 `sourceId` 不属于所选 `groupId`，结果允许为空，不做额外修正。

### 聚合展示规则

对命中的 item 按 cluster 组织后，按以下规则映射为前台条目：

1. 命中 item 没有 `clusterId`，直接显示为单条卡片。
2. 命中 item 有 `clusterId`，并且同 cluster 命中数大于 1，则显示为聚合卡片。
3. 命中 item 有 `clusterId`，但过滤后同 cluster 仅剩 1 条，则退化为单条卡片。

### 聚合卡片字段重算规则

聚合卡片的以下字段都以“命中的 item 子集”为准，而不是全量 cluster：

- `latestPublishedAt`
- `score`
- `sourceCount`
- `itemCount`
- `itemsPreview`
- `hasMoreItems`
- 展开后的条目列表

### 筛选命中语义

- 聚合内容只要存在至少一条相关内容命中筛选条件，就可以进入结果集。
- 但进入结果集后，仅展示命中的相关条目。
- 不会因为某个聚合中有一条命中，就把同聚合下未命中的内容一并展示出来。

这条规则是本次列表页改动的核心约束。

## 前台交互设计

### 后台设置页

在信息源管理区域新增两组交互：

1. 单源自动填充
   - 在新增信息源表单中增加“根据 RSS 自动填充”按钮。
   - 点击后调用解析 API，回填名称与站点 URL。
2. OPML 导入
   - 增加文件选择控件与导入按钮。
   - 导入成功后展示简短结果摘要并刷新页面。

编辑已有信息源暂不额外提供“重新解析”按钮，本次只覆盖新增和批量导入场景。

### 前台列表页

在当前“排序方式”和“时间范围”控制区新增：

- 分组下拉框
- 信息源下拉框

交互要求：

- 默认显示“全部分组”“全部信息源”。
- 选择分组后，信息源下拉框只显示该分组下的信息源；未分组信息源只在“全部分组”时可见。
- 切换分组或信息源时重新请求 feed API。
- 保持当前时间范围、排序和日期区间参数不丢失。

## 错误处理

### RSS 自动解析

- RSS URL 非法：返回 400。
- 远程订阅不可访问或解析失败：返回 422 或 502，并给出可读错误信息。

### OPML 导入

- 文件为空或非 XML：返回 400。
- 没有任何有效 `xmlUrl` 节点：返回 422。
- 单条订阅失败不终止整个导入，记录到结果摘要中，整体仍返回成功结果。

### 前台筛选

- 非法 `groupId` 或 `sourceId` 不报错，按无结果处理。
- 非法日期和排序仍沿用现有容错逻辑。

## 风险与缓解

### 风险 1：OPML 结构差异较大

不同导出工具的 OPML 属性名可能不完全一致。缓解方式是对常见属性 `text`、`title`、`xmlUrl`、`htmlUrl` 做宽松兼容，并在导入结果中记录无法识别的节点。

### 风险 2：筛选后聚合排序与展示不一致

如果继续以 cluster 原始字段排序，会与命中子集的最新时间或质量分不一致。缓解方式是在构造结果项时使用命中子集重新计算展示字段，并按展示字段完成最终排序。

### 风险 3：RSS 元信息解析依赖外部网络

测试中不应依赖真实网络。缓解方式是在 service 层注入可替换的解析器，在测试里使用 stub 返回固定 feed 元信息。

## 验收标准

- 后台可导入有效 OPML，并自动创建对应分组和信息源。
- 单个信息源新增时可根据 RSS URL 自动回填推荐名称和站点 URL。
- 列表页支持分组和信息源筛选。
- 聚合内容在筛选命中时只展示命中的相关条目。
- 聚合内容在筛选后只剩 1 条时，按单条内容样式展示。
- 现有时间范围、排序、抓取和审核功能行为不回退。
